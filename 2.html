next(ts) 로 gh-pages 에 blog 하기(1)

1. next 를 사용하는 이유?

react 는 강력한 기능을 가지고 있지만 그것을 다 사용하기에는 숙련된 경험이 필요합니다.

저는 백엔드 개발자이며 2015년 0.14x 버전의 리엑트를 사용한 경험이 전부입니다.
그 후에는 vue 를 1년여정도를 사용했지만 이때도 vue framework 인 nuxt 를 자주 사용했습니다.
라우터나 미들웨어 plugin 을 적용하는 방식이 손쉽게 다가왔습니다.
흡사 초기 2.5버전의 스프링을 설정할때 xml 지옥을 맛보다가
스프링부트가 나오고 통상적인 관례를 따지는 설정방식과 유사하다는 느낌을 받았습니다.

회사의 ui 라이브러리가 react 로 정해진점과
gatsby 같은 다른 프레임워크보다는 next 가 회사 업무의 연장으로 더 활용할수 있다 판단하였습니다.
그리고 마지막으로 제일 중요한 이유는 그냥 하고 싶었기 때문입니다.

2. next 시작하기

nuxt 는 버전이 좀 다르긴 해도 한글문서가 좀 잘 되어있고 설치시 cli 로 많은 starter 를 고르 수 있었는데
제가 못찾은건지 next 는 영문문서만 있고 cli 가 단순히 설치만 가능했습니다.

코드를 생성할 폴더로 가셔서 아래의 명령어를 수행해주세요.

2021-01-01 기준 node 10.13 이상 버젼이 설치되어있어야합니다.

npx create-next-app next-blog

(1) npx create-next-app next-blog 이 명령어를 수행하면 next-blog 라는 이름으로 next 프로젝트를 만들어줍니다.
1회성 작업이기 때문에 npx 로 수행했습니다.

npx 의 설명은 https://geonlee.tistory.com/32 해당 블로그에 잘 나와있으니 링크로 대신하겠습니다.
nextjs-blog 라는 폴더로 프로젝트가 생성이 됩니다. 원하시는 이름으로 생성하셔도 됩니다.

설치가 되고나면 아래와 같이 명령어를 수행해보라고 나옵니다.

yarn dev 는 npm run dev 와 같은 역할을 합니다.
yarn 은 npm 에 비해 몇가지 장점이 있는데요. 이부분은 검색하면 다양한 내용들이 있으니 찾아보길 권합니다.

cd next-blog
yarn dev

(1) next-blog 프로젝트 폴더로 이동합니다.
(2) yarn dev 를 수행합니다.
yarn dev 는 npm run dev 와 동일합니다.
package.json 의 script 에서 dev 에 적혀있는것을 수행하는 명령어입니다.
내부동작은 추후 next 시리즈를 연재할때 설명드리겠습니다.

명령어를 수행하고 나면 아래와 같은 결과가 나옵니다.

웹 브라우저에서 http://localhost:3000 을 수행해보세요.

그러면 아래와 같은 화면이 나올겁니다.

3. pages 설명

cli 명령을 작성하고 나면 아래와 같은 형태로 파일이 구성됩니다.

./pages/
ㄴ ./api/
  ㄴ hello.js
ㄴ _app.js
ㄴ index.js
./public/
ㄴ favicon.ico
ㄴ vercel.svg
./styles/
ㄴ Home.module.css
ㄴ global.css
.gitignore
README.md
package.json
yarn.lock

pages 폴더는 기본셋팅시 router 기능을 합니다.
./pages/api/hello.js 는 /api/hello 주소가 생기게 됩니다.

그래서 위의 최초 구성시
url 은 /api/hello 와 / 두개가 됩니다.

만약 ./pages/hello.js ./pages/hello/index.js 두개를 동시에 작성한다면
hello.js 로 라우팅이 되겠지만
warn - Duplicate page detected. pages/hello.js and pages/hello/index.js both resolve to /hello. 라는 로그가 발생합니다.

pages 폴더 밑에서 라우터기능을 하는건 참 매력적입니다.
별도의 라우터 설정을 안해도 되며 직관적이기 때문입니다.

그러면 블로그를 작성한다면 매번 이렇게 페이지를 생성해야할까요?
아닙니다. 우리는 마크다운파일을 생성해서 해당 파일의 이름(전 숫자로 할 예정)으로 동적 생성하게 작성할 예정입니다.
이건 다음 편에서 더 얘기를 드리겠습니다.

pages 에 _app.js 라는 파일이 궁금하실텐데 pages에 _ 로 시작되는 파일은 라우팅 대상이 아닙니다.
해당 파일은 nuxt 의 미들웨어와 비슷한 작업을 하게 됩니다. 이것도 다음에 어떻게 활용하는지 설명드리겠습니다.

4. ts 얹기

굳이 타입스크립트를 적용할 필요가 있을까? 궁금해 하실수 있는데요.
사실 블로그만 만드는데는 불필요합니다.
대신 저는 해당 블로그를 만들때 제 업무의 확장으로 보고 있어서
실제 업무로는 타입스크립트도 적용할거라 요것도 타입스크립트를 적용하려 합니다.

위에 뜬금없이 pages 설명한 이유는 불필요한 파일들을 삭제하기전에 왜 삭제해도 되는지 알려드리려는 떡밥이였습니다.

아까 말한 /api/hello 의 url 은 사용안할거니 아래의 두 파일을 삭제합니다.
./pages/api/hello.js
./pages/api/

이제 타입스크립트를 설정하기위해 명령어를 수행하겠습니다.

 touch tsconfig.json
 yarn add --dev typescript @types/react @types/node
 yarn dev

touch 는 windows 기본 커맨드창에서는 안될겁니다.
툴로 파일을 생성해도 됩니다.

(1) touch tsconfig.json 는 빈 내용의 tsconfig.json 파일을 만들어줍니다.
이름에도 알수 있듯이 json 형태의 typescript 설정파일입니다.

(2) typescript 를 적용하려면 최소 typescript @types/react @types/node 이렇게 3개의 라이브러리가 필요합니다.
yarn add --dev typescript @types/react @types/node 를 수행하게 되면
package.json 파일에 devDependencies 로 3개의 라이브러리가 추가된것을 확인할수 있습니다.

(3) yarn dev 이렇게 수행하면 아래와 같은 메시지가 나오면서 서버가 시동됩니다.

아까와는 다르게 detected typescript 라고 나오는것을 볼수 있습니다.
파일 목록을 조회해보면 next-env.d.ts 파일이 새로 생성이 되었습니다.

next-env.d.ts 파일은 링크 에 가보시면 다음과 같이 써져있습니다.
A file named next-env.d.ts will be created in the root of your project. This file ensures Next.js types are picked up by the TypeScript compiler. You cannot remove it, however, you can edit it (but you don't need to).

아까 만든 tsconfig.json 파일에 들어가면 내용이 채워져있을겁니다.
tsconfig.json 에 strict 모드는 기본이 false 입니다.
true로 변경해주세요.

그리고 typescript를 적용할 파일들을 tsx 로 변경해줍니다.
./pages/index.js -> ./pages/index.tsx
./pages/_app.js -> ./pages/_app.tsx
로 이름을 변경해줍니다.

ts 와 tsx 확장자가 있는데 jsx 문법이 적용되어있으면 tsx 로 변경해주시고 그 외에는 ts로 변경해주시면 됩니다.

파일이름이 변경되었기때문에 yarn dev 로 다시 재시작하고 브라우저를 새로고침 하겠습니다.

잘 나오시나요?
